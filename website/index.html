Architecture

One proxy. Every query. Zero shared credentials.

Nobody gets the database password — not even your own team. AccessFlow sits in the middle and is the only thing that ever connects to the real database, and only after a query is approved. Users sign in to AccessFlow, never to the database; behind the scenes the proxy holds the real database connection pools and opens connections on their behalf.

Full query proxy — SQL and NoSQL

Every query is inspected before it ever reaches the database. Each statement is parsed, classified as SELECT / DML / DDL, and validated against schema and table allow-lists at the query-structure level, then routed through your review policy before it touches the database. MongoDB gets the same treatment through a native engine — shell (db.coll.find({…})) or JSON-command queries are parsed, classified, and governed identically, with row-level security applied as $match stages and field masking on returned documents; results render as both a JSON document view and a flattened table. Couchbase joins the NoSQL line-up with SQL++ (N1QL): statements are classified like SQL, dangerous constructs (CURL(), JavaScript UDFs, system:* keyspaces) are rejected up front, and row-level security is ANDed into the WHERE clause with parameter binding — never string concatenation. Redis brings governed key-value access: redis-cli commands (GET, HGETALL, SCAN, SET, DEL) are classified onto the same workflow, server-side scripting and blast-radius commands (EVAL, CONFIG, FLUSHALL) are rejected up front, field masking applies to returned values, and row-security policies fail closed. Cassandra and the CQL-compatible ScyllaDB add wide-column governance through a single native engine: CQL statements are classified like SQL (SELECT / INSERT / UPDATE / DELETE, plus CREATE/ALTER/DROP and TRUNCATE), server-side code (BEGIN … BATCH, CREATE FUNCTION/AGGREGATE) is rejected up front, and row-level security is key-aware and fails closed — predicates splice into the WHERE clause only on partition/clustering key columns with CQL-filterable operators, never injecting ALLOW FILTERING. Elasticsearch and the wire-compatible OpenSearch bring governed search access through a single native engine: a JSON query envelope ({"search":"logs-*","query":{…}}, plus count, get, index/bulk, update/delete-by-query, and index management) is classified onto the same workflow, server-side scripting (script, runtime_mappings, Painless) and cluster APIs are rejected up front, row-level security injects bool.filter clauses on keyword fields (fail-closed on writes into a policied index), and field masking applies recursively to _source fields including nested dot-paths. Amazon DynamoDB brings governed key-value access through PartiQL: SELECT / INSERT / UPDATE / DELETE statements (and JSON table-management commands) are classified onto the same workflow, transaction/batch statements are rejected, row-level security splices predicates into the WHERE clause with parameter binding on any attribute, and field masking applies recursively by dot-path — and its connection is cloud credentials + region rather than host/port, the first engine to exercise that. Neo4j brings governed graph access through Cypher over Bolt: clause-based statements classify onto the same workflow by their strongest write clause (DELETE / CREATE-MERGE / SET / read), LOAD CSV and procedure calls outside a read-only allow-list are rejected up front, row-level security ANDs property predicates onto each MATCH's WHERE with parameter binding (node-label policies, fail-closed on anonymous or write shapes), and field masking is label-aware and recursive — connecting via host/port or a full bolt:// / neo4j+s:// URI (Neo4j Aura / clustered routing). Optional read-replica routing ships SELECT traffic to a sibling pool while writes always hit the primary — with automatic primary fall-back and an audit row on replica failure. Dynamic data masking renders sensitive columns per requester — full, partial, hash, email, or format-preserving — with role / group / user reveal conditions, applied before results are ever serialized or stored. Row-level security injects per-table predicates into the parsed query so a scoped user only sees or affects the rows they are authorised for — values bound as parameters, never concatenated; un-rewritable queries rejected rather than run unfiltered.

AI query analysis

An AI assistant reads each query first and flags anything risky — like a senior reviewer who never sleeps. The analyzer is pluggable per organization — Anthropic, OpenAI, Ollama, any OpenAI-compatible backend (vLLM, LM Studio, Together, Groq, OpenRouter, …), or Hugging Face (hosted Inference Providers router or a local / self-hosted TGI server). Every query gets a 0–100 risk score, missing-index detection, and anti-pattern flags before a human ever sees it. Beyond flagging problems it proposes concrete fixes — dialect-aware index recommendations and query rewrites, each with a one-click “Apply as draft” that pre-fills the editor and routes the suggestion through the normal review pipeline (nothing auto-executes). Tailor the analyzer's behaviour with an editable system prompt per configuration — override the built-in template or leave it blank to use the default. Opt in per datasource to text-to-query — users describe what they want in plain language and the AI drafts a schema-grounded query into the editor, in the engine's native query language (SQL, plus MongoDB shell/JSON, Cypher, CQL, Elasticsearch Query DSL, redis-cli, SQL++, PartiQL for the NoSQL engines; restricted fields are never referenced); the draft still flows through risk analysis and human review. Attach a per-configuration RAG knowledge base — knowledge documents are embedded and the most relevant chunks are retrieved into the prompt so analysis reflects your house rules (pluggable vector store: in-app pgvector or external Qdrant). Optional Langfuse integration adds LLM tracing of every analysis and runtime prompt management. An admin /admin/ai-analyses dashboard charts average risk over time, top issue categories, and most active submitters.

Configurable review workflows

Decide who must sign off before a query runs, and route each request automatically. Build per-datasource approval chains with multi-stage reviewers, auto-approve reads, approval timeouts that auto-reject, and submitter cancellation. Bulk approve, reject, or request changes on whole batches from /reviews so high-volume queues clear in one action. Scope reviewers to specific datasources by user or by group so each team only sees its own queue. Layer on policy-as-code routing — ordered rules that, right after AI analysis, auto-approve, auto-reject, require N approvals, or escalate a query based on its type, referenced tables, AI risk, requester role or group, and time-of-day; first match by priority wins, anything unmatched falls through to the review plan. Users can also raise a just-in-time access request — temporary, scoped read/write/DDL access for a set duration that runs through the same approval engine, is granted as a time-boxed permission, and is auto-revoked on expiry. Reviewers are paged via Email, Slack, Discord, Telegram, Microsoft Teams, PagerDuty, or HMAC-signed webhooks — and with a Slack app connected they can approve or reject directly from the Slack message, with the same self-approval and role checks as the UI. While a query sits in review, the submitter and reviewers can co-author it in real time — live presence, remote cursors, conflict-free merge, and inline comment threads anchored to the SQL — and any edits re-enter the workflow through the normal submit path, so approvals are never bypassed.

Workforce-ready auth

Sign in with the accounts your company already uses. JWT RS256 with HttpOnly refresh-token rotation. SAML 2.0 SP- and IdP-initiated SSO with attribute-based role mapping and per-IdP group mapping that auto-syncs AccessFlow group memberships on every login. OAuth 2.0 / OIDC sign-in with built-in templates for Google, GitHub, GitHub Enterprise Server, Microsoft, GitLab, and self-managed GitLab, plus a generic OIDC provider for other IdPs (Keycloak, Auth0, Okta, Authentik, Zitadel) — plus optional TOTP 2FA. One deployment hosts multiple fully-isolated organizations (every entity is scoped by org, always derived from the JWT — never the request body); a super-admin (platform admin) manages tenants across the cluster — create, edit, disable / enable — with per-org quotas (datasources, users, queries-per-day; a breach returns 409 QUOTA_EXCEEDED) and a disabled-org kill-switch that blocks login and requests immediately.

JWT · SAML · OAuth · IdP group sync · TOTP · Multi-tenant orgs · Per-org quotas

Tamper-evident audit

Every decision is recorded in a log that can't be quietly edited after the fact. The append-only metadata audit log is chained with HMAC-SHA256, and INSERT-only DB grants make after-the-fact rewrites detectable. CSV export, searchable UI, no row data ever stored.

Cloud-native deploy

Run it on a laptop with one command, or on Kubernetes for production. One docker compose up covers local and small environments; a Helm 3 chart covers Kubernetes, with HPA, PDB, and Bitnami Postgres/Redis subcharts. Redis-backed distributed locks keep scheduled jobs safe across replicas.

Connectors

A growing catalog of SQL and NoSQL connectors.

A connector is simply how AccessFlow talks to one kind of database. AccessFlow ships with a built-in list — find the database your team uses, click Install, and it's ready to govern; AccessFlow fetches the right driver for you and verifies it's genuine before it's ever used. The list is grouped into SQL (table-based databases like PostgreSQL) and NoSQL (document databases like MongoDB). Using something that isn't on the list? Upload its driver and AccessFlow adds it. Under the hood this is a versioned connector catalog: every driver download is SHA-256-verified and cached, and the MongoDB, Couchbase, Redis, Cassandra, ScyllaDB, Elasticsearch, OpenSearch, Amazon DynamoDB, and Neo4j native engines are resolved on demand the same way. Request flow

From keystroke to commit, observable at every step.

Every query follows the same predictable path — submitted, reviewed, run — and each step is recorded. A single state machine drives that flow through classification, review, and execution, and every transition is auditable, cancellable, and replayable.

01

Submit a query

Analyst writes SQL in the built-in CodeMirror editor. The frontend posts to /queries; the proxy parses and classifies it.

02

AI reviews it

Risk score, anti-pattern flags, missing indexes. The status moves PENDING_AI → PENDING_REVIEW.

03

Humans approve

Reviewers get paged by Email, Slack, Discord, Telegram, Microsoft Teams, PagerDuty, or webhook. Multi-stage approval chains run as a state machine.

04

Proxy executes

AccessFlow opens a connection from the per-datasource connection pool, executes, captures metadata.

05

Audit forever

Append-only metadata audit row. WebSocket pushes result to the submitter. Status becomes EXECUTED.

Quick start

Up and governing queries in under five minutes.

Pick your platform. The backend initializes its database schema on first boot; the first admin is created through the in-app setup wizard the moment you open the SPA.

docker-compose.yml 
# Zero-config demo — open http://localhost:5173 and follow the setup wizard.
# Demo-only JWT / encryption keys are embedded in the file; generate your own
# for anything beyond evaluation (see docs/09-deployment.md).
services:
  postgres:
    image: postgres:18
    environment:
      POSTGRES_DB: accessflow
      POSTGRES_USER: accessflow
      POSTGRES_PASSWORD: accessflow

  redis:
    image: redis:8-alpine

  backend:
    image: ghcr.io/bablsoft/accessflow-backend:latest
    depends_on: [postgres, redis]
    environment:
      DB_URL: jdbc:postgresql://postgres:5432/accessflow
      REDIS_URL: redis://redis:6379
      ENCRYPTION_KEY: # 64-char hex, demo default embedded
      JWT_PRIVATE_KEY: # PKCS#8 PEM, demo default embedded
    ports: ["8080:8080"]

  frontend:
    image: ghcr.io/bablsoft/accessflow-frontend:latest
    depends_on: [backend]
    ports: ["5173:80"]
values.yaml 
# helm repo add accessflow https://bablsoft.github.io/accessflow
# helm install accessflow accessflow/accessflow -n accessflow --create-namespace
# Encryption key, JWT key, and Postgres password are auto-generated on first install
# and preserved across upgrades — no Secrets need to be pre-created.
#
# Ready-made starting points for common shapes live under
#   charts/accessflow/examples/   (minimal, production, external-services, bootstrap, airgapped)

replicaCount:
  backend: 2
  frontend: 2

# Postgres + Redis ship as Bitnami subcharts — toggle off for managed services
postgresql:
  enabled: true
redis:
  enabled: true

# HPA + PDB are off by default — turn on for production
autoscaling:
  backend:
    enabled: false
    minReplicas: 2
    maxReplicas: 10

# Override only if you want to manage secrets externally
# (sealed-secrets, External Secrets, Vault, …):
# config:
#   encryptionKey: { existingSecret: my-secret, key: value }
#   jwtPrivateKey: { existingSecret: my-secret, key: jwt }

ingress:
  enabled: true
  className: nginx
  hosts:
    - host: accessflow.company.com
  # TLS is off by default — turn on and supply a Secret to terminate HTTPS
  tls:
    enabled: false
shell 
# clone
git clone https://github.com/bablsoft/accessflow.git
cd accessflow

# 1. infrastructure — Postgres 18 + Redis 8 + Mailcrab (dev-only compose)
docker compose -f backend/docker-compose-dev.yml up -d

# 2. backend — Java 25 · Spring Boot 4 · Maven
cd backend
./mvnw spring-boot:run

# 3. frontend — Node 24 · Vite 8 · React 19
cd ../frontend
npm install
npm run dev

# open http://localhost:5173 — the in-app setup wizard creates the first admin
